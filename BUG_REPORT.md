# FureIn E2E QA Test Report - May 2026

## Executive Summary
Comprehensive end-to-end testing was performed using the `agent-browser` tool. While the core shopping discovery and cart flows are functional, several critical architectural and implementation bugs were identified in the checkout and authentication/dashboard layers.

---

## 🟢 Functional Features
- **Database Connection:** Prisma correctly interacts with PostgreSQL. Schema sync and seeding are functional.
- **Homepage:** All sections (Hero, Categories, Trending) load correctly with real data from the database.
- **Shop & Discovery:** 
  - Product listing page (PLP) works.
  - Live search filtering works as expected.
  - Category filtering works visually.
- **Product Details Page (PDP):** Correctly fetches product data by slug, displaying galleries and specifications.
- **Cart Management:** Adding items to the cart works correctly and persists in the Zustand store.

---

## 🔴 Critical Bugs & Issues

### 1. Checkout Redirection Bug
- **Symptom:** Navigating to `/checkout` (either via button or direct URL) immediately redirects the user back to `/shop`.
- **Root Cause:** In `allecom/components/checkout/checkout-client-content.tsx`, a `useEffect` redirects to `/shop` if `items.length === 0`. Because Zustand's persisted state hydrates *after* the initial mount, the items array is empty on the first tick, triggering the redirect before the cart can hydrate.
- **Severity:** High (Blocks checkout).

### 2. Auth & Dashboard Sync Failure
- **Symptom:** After registering or logging in, the user is redirected to the homepage. Attempting to access `/dashboard` redirects the user back to `/login`, despite having an active `next-auth` session.
- **Root Cause:** The `DashboardLayout` checks for a user in the `useUserStore` (Zustand). However, the `LoginForm` and `RegisterForm` only perform `signIn` via `next-auth` and **never update the Zustand store**. This creates a "split personality" where the UI thinks the user is logged out while the session is actually active.
- **Severity:** High (Blocks user dashboard access).

### 3. Database Utility Error
- **Symptom:** `npx tsx check-db.ts` fails with `Error: SASL: SCRAM-SERVER-FIRST-MESSAGE: client password must be a string`.
- **Root Cause:** The script likely fails to load the `.env` file when run via `tsx` without a dotenv wrapper, causing the `DATABASE_URL` (specifically the password part) to be parsed incorrectly or seen as undefined in the `pg` pool.
- **Severity:** Low (Utility script only; Prisma CLI works fine).

---

## 🛠️ Recommendations
1.  **Checkout Fix:** Add a `_hasHydrated` flag to the Zustand store or a delay/check in `CheckoutClientContent` to prevent premature redirects.
2.  **Auth Fix:** Synchronize `next-auth` sessions with the `useUserStore`. Ideally, use `useSession` from `next-auth` exclusively for auth state to avoid redundancy.
3.  **Utility Fix:** Update `check-db.ts` to use `dotenv` explicitly or run it with `tsx -r dotenv/config`.
